## 2. Key findings & caveats

### 2.1 Precision: yfinance is float64 — never trust the raw repr
yfinance returns prices as IEEE-754 **float64** for every market (e.g. RM2.32 comes back
`2.3199999332…`). `max_decimals` on yfinance therefore reflects float noise, **not** market
tick precision. Consequences for `pricing/`:
- Always convert via `Decimal(str(value))` and **quantize at ingestion** to the market's
  tick (US 2dp · TW tick-dependent · **MY up to 3dp**). Never persist the float repr.
- For **MY 3-dp / sub-RM1 tick fidelity**, the authoritative sources are the **string**
  ones: **klsescreener** (`#price` `data-value="2.260"`) and **Malaysiastock.biz** (3-dp).
  TWSE/TPEx return strings too (`'2,295.00'`). Use a string source to cross-check yfinance
  for tick-sensitive counters. (Our MY samples are all ≥RM1, so the 0.005/0.001 sub-RM1
  band wasn't directly exercised — confirm with a sub-RM1 counter at ingestion time.)

### 2.2 TW board classification must be resolved per instrument
`6531` and `6139` were missed by **both** yfinance `.TWO` **and** the TPEx mainboard
endpoint (yfinance logged "possibly delisted"). The initial 上市/上櫃 guess was wrong for
some codes — board membership (TWSE 上市 · TPEx 上櫃 · 興櫃 emerging) is **not guessable**.
`pricing/` must **store the resolved market/board + suffix per instrument** and verify
coverage at ingestion; the "possibly delisted"/empty-result signal is a coverage alarm,
not a silent miss. (Confirmed working: `8299`, `6488` on TPEx; `0050/2454/2330/2543` on TWSE.)

### 2.3 Source reliability / latency
- **stockprices.dev**: worked for AAPL (307.34) but returned `400` for TSLA and `429` on a
  follow-up despite "no limits" marketing → **fallback only, latest-only, no history**.
- **klsescreener scrape**: ~3.5 s per code (per-page) vs yfinance batch (sub-second
  amortized) → use for **corroboration / gap-fill**, not bulk pulls.
- **bursamalaysia.com**: Cloudflare `403` JS-challenge → not scrapable with plain
  `requests` (would need a headless browser); deprioritized.

## 7. Recommended `pricing/` architecture (input to its spec)

- **Provider protocol** — one adapter per source implementing the subset it supports:
  `fetch_quote_latest`, `fetch_quote_history`, `fetch_fx`, `fetch_dividends`, returning
  normalized rows: `Decimal` (via `str`), as-of date, and a **`source` tag** (provenance).
- **Registry + config-driven ordered chain** per `(data_type, market)`, **runtime-adjustable**
  (settings page; FinMind token + per-source order live there).
- **Fallback chain** — try sources in order until one returns valid data; record the
  winning source per row.
- **Graceful degradation** — on miss/stale, serve last-known with a **staleness flag**;
  never fabricate, never crash the dashboard.
- **Precision** — `Decimal(str(x))` everywhere; store at source precision, quantize at
  display; for tick-sensitive MY counters prefer/cross-check a string source.
- **Per-instrument board/suffix resolution** stored on the instrument (TW 上市/上櫃/興櫃;
  `.KL` for MY) and verified at ingestion (coverage alarms, not silent misses).
