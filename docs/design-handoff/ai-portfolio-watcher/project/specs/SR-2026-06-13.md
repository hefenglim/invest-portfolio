## 同場補強（非衝突，消除模糊）

- **`fee_snapshot` 存 raw 值非顯示字串**（spec 11）：DB 存 `FeeRuleSet` raw Decimal＋`label`；
  前端只渲染 label。理由：spec 02 對帳匯出需 raw，顯示字串落庫會造成雙重事實。
  ⚠ 前端接線時 ledger.js 的逐 key 顯示需小改（已註記）。
- **ETF 判定單一來源**（spec 10/12）：`instruments.is_etf` 新欄位；禁止以 `sector=="ETF"` 推導。
- **時區規則**（spec 08 §8.0）：DB 存 UTC；API 回應一律 ISO-8601 含偏移、固定 `Asia/Taipei`。
- **`GET /api/alerts` 定位**（spec 03）：與 dashboard 內嵌 `alerts` 必須共用同一計算函式；
  此 endpoint 僅供非儀表板頁鈴鐺輕量刷新。
- **分頁慣例**（spec 08 §8.0）：`limit/offset`＋`total_count`，limit 上限 500。
- **spec 01 補 `partial` 欄位**（回補逾時的部分回傳，原只在實作要點提及未入契約）。

## ✅ Q1 — 已定案關閉（2026-06-13，使用者授權架構師決策）

**同一標的多帳戶持有時的歸屬規則 — 採暫行定案為正式契約：**

1. 個股抽屜（spec 01）：`cost_basis` 綁定單一帳戶持倉列，預設＝**該 symbol 持股最多的帳戶**，
   `cost_basis.account_id` 回聲實際採用帳戶，抽屜 badge 顯示帳戶名（現行 detail.js 已如此）。
2. 買賣試算（spec 03 §3.2）：`account_id` 選填，缺省同上規則＋回應回聲。
3. 再平衡試算（spec 03 §3.3）：賣出分配同一規則（先賣持股最多帳戶，不跨帳戶拆單）。

日後若出現多帳戶同標的的真實需求，升級路徑＝抽屜加帳戶切換 tabs（純前端增量，
API 已有 account_id 參數與回聲，契約不變）。**本項不再阻塞任何開發。**
