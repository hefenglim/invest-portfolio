「有什麼新功能」 — feature-announcement system

WP-WN · v0.1.18 candidate · 2026-07-13. A dismissible new-feature badge plus a per-version panel whose entries deep-link (「前往」) to where each feature lives. Front-end only for the badge/panel; a tiny single-row table persists the acknowledged version server-side.

Goal

  • Tell a returning user, at a glance, that new features shipped — a small accent dot on the topbar ✦ button and a NEW pill by the sidebar version tag when there are unseen features.
  • A ✦ panel lists each visible version's features (title · one-line desc · area breadcrumb) with a 「前往 →」 button that jumps to the page/tab the feature lives on.
  • Opening the panel acknowledges everything up to the current version; the badge disappears and the acknowledgement persists across reloads and devices (server-side, not localStorage).

API contract — /api/whats-new

Global session gate covers it (guest open, protected 401). Counts + strings only; no money. The router is thin — ordering/visibility live in shared/whatsnew.py.

RouteBehaviour
GET /api/whats-new Returns current_version, seen_version, unseen_count, and versions[] (newest first). Each group: version, date (nullable), unseen, features[] (id/title/desc/href/area).
POST /api/whats-new/seen Body {version}. Bad format → 400 (error.field == "version"). Else acknowledges (monotonic) and returns the same payload shape as GET, so the client re-renders from one response. Idempotent.
{
  "current_version": "0.1.17",
  "seen_version": "0",
  "unseen_count": 15,
  "versions": [
    { "version": "0.1.17", "date": "2026-07-13", "unseen": true,
      "features": [ { "id": "market-risk-alerts", "title": "市場風險預警",
        "desc": "…", "href": "settings.html#alerts", "area": "系統設定 → 預警規則" } ] }
  ]
}

Seen-state semantics (version-level, monotonic)

  • Single-row table whatsnew_config(id=1, seen_version, updated_at) via the same config_store create-always / seed-once pattern as ui_prefs. Default seen_version = "0" (below every real version, so a fresh install sees everything as new).
  • Acknowledgement is version-level, not per-feature: opening the panel marks everything up to current_version as seen. A group is unseen iff version_key(version) > version_key(seen_version); unseen_count sums the features in unseen groups.
  • Monotonic: set_seen_version keeps the max of stored vs supplied (by numeric version key), so a replayed or out-of-order lower version can never un-acknowledge newer features.
  • version_key parses to a numeric tuple so 0.1.9 < 0.1.10 orders correctly (the lexical trap). Versions validate against ^\d+(\.\d+)*$.

Catalog maintenance rule

  • CATALOG in shared/whatsnew.py is static and hand-maintained, newest version first, 2–4 user-facing features per version (what the user can now do, not implementation detail). VERSION_DATES maps version → CHANGELOG date.
  • visible_versions(current) filters to versions ≤ current, newest first, capped at _MAX_VERSIONS = 6. The v0.1.18 catalog entry therefore stays hidden until __version__ is bumped at ship time — intentional and test-pinned.
  • Ship-step durability: the ship-version checklist (item 5b) requires adding the new version's CATALOG + VERSION_DATES entry. Unit tests enforce: ids unique per version, valid version format, every href page exists under web/ (and a settings.html#tab hash is a real tab id), and every catalog version ≤ __version__ has a ## [v…] CHANGELOG heading.

Front-end decisions

  • Dynamic load, no HTML tag. whatsnew.js is lazy-loaded by shell.js after the /api/health version resolves, and stamped ?v=<version> by shell (the asset stamper only sees HTML tags). This avoids touching every web/*.html.
  • Silent degradation everywhere. Every network path (badge fetch, ack POST) tolerates failure without breaking the page — the badge is a hint, not a gate.
  • Optimistic ack. Opening the panel removes the dot/pill immediately and fires one POST seen per open; a failed POST does not re-show the badge.
  • 前往 navigation. Same page → set location.hash (the settings page's own hashchange handler switches tabs) and flash the target in place; other page → location.href, with a sessionStorage flash marker consumed on arrival. The arrival flash respects prefers-reduced-motion.

Round 2 (field feedback) — 2026-07-13

Three owner-requested UX upgrades to the 前往 arrival experience. Backend adds only presentation metadata; all behaviour is front-end.

  • A · 10s flash, cancel-on-switch, fire only via 前往. (The 10s hold-then-fade timing is superseded by the Round 3 blink; the cancel-on-switch / fire-only-via-前往 behaviour below is unchanged.) The wn-flash-pulse keyframes now hold the ring + tint clearly visible to ~8.5s then fade over the last ~1.5s (single 10s iteration; JS safety-net ~11s; reduced-motion fallback kept). While a flash/callout is active a hashchange (and the settings pd-settings-tab tab-switch event, which uses replaceState and fires no hashchange) tears it down immediately; teardown removes the callout from the DOM so it never resurfaces on switch-back — only a fresh 前往 re-arms it. The sessionStorage marker now carries a ts; a marker older than 30s is discarded unconsumed, and it is always removed once evaluated on the matching page.
  • B · In-page detailed feature callout (頁內新功能提醒). Arriving via 前往 now also drops a dismissible .wn-callout card inserted before the anchor (normal flow — pushes content down, never overlays controls): accent left-border, ✦ 新功能 + title, desc body, area breadcrumb, ✕ close. The marker became a JSON string {href,id,title,desc,area,target,ts} (malformed/legacy markers discarded silently). The anchor resolves from the feature's target selector (small/inline matches resolve to el.closest('section.panel')), falling back to the settings #view-<hash> section. Same-page 前往 runs the SAME routine (callout + scroll + 10s flash), not a bare flash. One callout at a time (re-arrival replaces it).
  • Model. Feature gained target: str | None — an optional CSS selector, serialized in the GET/POST feature JSON. Seeded: market-risk-alerts → #alert-rules-wrap, target-weights → #target-weights-panel, the five notification-channel features → .nt-cards, quiet-hours → #nt-qh-enabled, per-rule-subscriptions → #nt-subs; instruments/insights/index features stay null. Presentation metadata only — no logic in shared/whatsnew.py. Contract/unit tests pin the seeded targets and the target ⇒ href invariant; an e2e test covers callout arrival + cancel-on-switch + no-resurface.
  • C · "missing NEW pill" report — no code change. Root-caused to the demo DB's seen_version already acknowledged by earlier verification runs; badge logic is correct and left intact.

Round 3 (owner UX round) — 2026-07-13

Four owner-requested changes. Per-feature read semantics replace the version-level ack; a full version-history browser is added; the arrival flash becomes a blink; the ✦ glyph becomes a crisp SVG.

  • 1 · Per-FEATURE seen semantics (supersedes the version-level ack above). A feature's NEW clears ONLY when the user acts on it — 前往 (navigate), 知道了 (dismiss, for href-less features), or 全部標示已讀 (whole visible window). Opening the panel no longer acknowledges anything. New table whatsnew_seen(feature_key PRIMARY KEY, seen_at), keyed "<version>:<id>", created alongside the legacy whatsnew_config by the same config_store create-always pattern. A one-time, idempotent migration folds a legacy seen_version > "0" into per-feature rows (INSERT OR IGNORE for every catalog feature at/below it); the legacy table is kept in place, never dropped (a legacy-schema boot crash is a known repo lesson) and simply stops being written. Shared API: get_seen_keys, mark_seen(keys) (idempotent, returns the full set), known_feature_keys(current) (visible-window keys, used to validate POSTs). get/set_seen_version were removed.
    Router. GET each feature gains key + seen; the group unseen is any-unseen; unseen_count counts unseen features across visible groups; seen_version is dropped. POST /seen body is now {features:[key,…]} OR {all:true} (all = every key in the visible window); an unknown/out-of-window key → 400 validation_error (field=features), nothing written; idempotent; returns the GET payload.
    Front-end. Ambient badge shows iff unseen_count > 0. Panel rows carry a per-feature NEW mini-pill; the group-head pill shows iff any row is unseen. 前往 marks that feature seen optimistically (remove the row pill, recompute group pill + badge from the cached payload) then POSTs; same-page 前往 and 知道了 are fire-and-forget, but cross-page 前往 awaits the POST (≤1.5 s cap) before navigating so the seen-state reliably survives the page unload. 全部標示已讀 clears every pill + the badge. Server truth reconciles on reload.
  • 2 · Blink pattern (replaces the 10s hold-then-fade). The wn-flash-pulse keyframes are now 0%/100% fully transparent, 50% full highlight (ring + soft tint), driven by animation: wn-flash-pulse 2s ease-in-out 10 — 1s in / 1s out, ten times (~20 s). JS safety-net FLASH_MS bumped to ~21 s; prefers-reduced-motion → none and cancel-on-switch are unchanged.
  • 3 · 版本發佈資訊 — full version-history browser (settings 一般). The whatsnew CATALOG is now the single user-facing history source: entries were backfilled for every shipped version v0.1.0 → v0.1.11 from CHANGELOG (1–4 zh-TW user-facing features each; href=None, since the ✦ panel caps at the newest 6 versions and these surface only here). New GET /api/whats-new/history?offset&limit pages the FULL catalog (version ≤ current, newest first): {total, offset, versions:[{version, date, features:[{title, desc, area}]}]}, offset ≥ 0, 1 ≤ limit ≤ 20 (else 400), no seen-state. A settings-page button lazy- ensures whatsnew.js (new window.pdEnsureWhatsNew hook) and calls window.pdWhatsNew.openHistory(); the modal (own .wnh-* namespace) appends each loaded page, with 「載入更早版本」 hiding when offset+len ≥ total. Scalability: the browser only renders the pages it loaded, so an ever-growing catalog can never degrade page performance (noted in the router + shared module docstrings). A full-coverage drift unit test now also asserts every CHANGELOG ## [v…] version (≥ 0.1.0, ≤ current) has a catalog entry.
  • 4 · Crisp ✦ icon. The blurry ✦ text glyph in #wn-btn is replaced by an inline four-pointed sparkle SVG (viewBox 0 0 24 24, fill=currentColor, 15×15, aria-hidden, display:block to centre it against the neighbouring topbar glyph buttons); the absolutely-positioned .wn-dot overlay is unaffected.

Round 4 — flash retiming + arrival unification (2026-07-14)

  • Blink retimed. The arrival pulse is now animation: wn-flash-pulse 1s ease-in-out 20 — 0.5 s fade-in / 0.5 s fade-out per 1 s cycle, twenty cycles (~20 s total). Keyframes are unchanged (0%/100% transparent, 50% full highlight); the JS safety-net FLASH_MS stays ~21 s. Replaces round-2's 2s … 10 (1 s in / 1 s out) — same duration, gentler per-cycle motion.
  • Every 前往 now flashes. _resolveAnchor gained a final generic fallback: after the precise target selector and the settings-tab / hash lookup miss, it returns document.querySelector('.page section.panel'), else document.querySelector('.page'). So a catalog feature whose page has no hash section (instruments / insights / index) still gets the callout + a page-level highlight instead of silently showing nothing.
  • Targets filled + enforced. Every CATALOG feature that carries an href now also carries a stable target (attribute selectors like section[data-screen-label="持倉明細"]). A new unit test asserts href ⇒ non-empty target forever, and the ship-version checklist (item 5b) now requires both for every shipped user-facing feature.
