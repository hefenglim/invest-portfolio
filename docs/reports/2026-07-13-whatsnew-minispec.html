「有什麼新功能」 — feature-announcement system

WP-WN · v0.1.18 candidate · 2026-07-13. A dismissible new-feature badge plus a per-version panel whose entries deep-link (「前往」) to where each feature lives. Front-end only for the badge/panel; a tiny single-row table persists the acknowledged version server-side.

Goal

  • Tell a returning user, at a glance, that new features shipped — a small accent dot on the topbar ✦ button and a NEW pill by the sidebar version tag when there are unseen features.
  • A ✦ panel lists each visible version's features (title · one-line desc · area breadcrumb) with a 「前往 →」 button that jumps to the page/tab the feature lives on.
  • Opening the panel acknowledges everything up to the current version; the badge disappears and the acknowledgement persists across reloads and devices (server-side, not localStorage).

API contract — /api/whats-new

Global session gate covers it (guest open, protected 401). Counts + strings only; no money. The router is thin — ordering/visibility live in shared/whatsnew.py.

RouteBehaviour
GET /api/whats-new Returns current_version, seen_version, unseen_count, and versions[] (newest first). Each group: version, date (nullable), unseen, features[] (id/title/desc/href/area).
POST /api/whats-new/seen Body {version}. Bad format → 400 (error.field == "version"). Else acknowledges (monotonic) and returns the same payload shape as GET, so the client re-renders from one response. Idempotent.
{
  "current_version": "0.1.17",
  "seen_version": "0",
  "unseen_count": 15,
  "versions": [
    { "version": "0.1.17", "date": "2026-07-13", "unseen": true,
      "features": [ { "id": "market-risk-alerts", "title": "市場風險預警",
        "desc": "…", "href": "settings.html#alerts", "area": "系統設定 → 預警規則" } ] }
  ]
}

Seen-state semantics (version-level, monotonic)

  • Single-row table whatsnew_config(id=1, seen_version, updated_at) via the same config_store create-always / seed-once pattern as ui_prefs. Default seen_version = "0" (below every real version, so a fresh install sees everything as new).
  • Acknowledgement is version-level, not per-feature: opening the panel marks everything up to current_version as seen. A group is unseen iff version_key(version) > version_key(seen_version); unseen_count sums the features in unseen groups.
  • Monotonic: set_seen_version keeps the max of stored vs supplied (by numeric version key), so a replayed or out-of-order lower version can never un-acknowledge newer features.
  • version_key parses to a numeric tuple so 0.1.9 < 0.1.10 orders correctly (the lexical trap). Versions validate against ^\d+(\.\d+)*$.

Catalog maintenance rule

  • CATALOG in shared/whatsnew.py is static and hand-maintained, newest version first, 2–4 user-facing features per version (what the user can now do, not implementation detail). VERSION_DATES maps version → CHANGELOG date.
  • visible_versions(current) filters to versions ≤ current, newest first, capped at _MAX_VERSIONS = 6. The v0.1.18 catalog entry therefore stays hidden until __version__ is bumped at ship time — intentional and test-pinned.
  • Ship-step durability: the ship-version checklist (item 5b) requires adding the new version's CATALOG + VERSION_DATES entry. Unit tests enforce: ids unique per version, valid version format, every href page exists under web/ (and a settings.html#tab hash is a real tab id), and every catalog version ≤ __version__ has a ## [v…] CHANGELOG heading.

Front-end decisions

  • Dynamic load, no HTML tag. whatsnew.js is lazy-loaded by shell.js after the /api/health version resolves, and stamped ?v=<version> by shell (the asset stamper only sees HTML tags). This avoids touching every web/*.html.
  • Silent degradation everywhere. Every network path (badge fetch, ack POST) tolerates failure without breaking the page — the badge is a hint, not a gate.
  • Optimistic ack. Opening the panel removes the dot/pill immediately and fires one POST seen per open; a failed POST does not re-show the badge.
  • 前往 navigation. Same page → set location.hash (the settings page's own hashchange handler switches tabs) and flash the target in place; other page → location.href, with a sessionStorage flash marker consumed on arrival. The arrival flash respects prefers-reduced-motion.
